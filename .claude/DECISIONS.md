## 2026-07-20 — v0.2 release hardening: all packages to 0.2.0, README as two co-equal deployment modes, synthetic signing-secrets allowlisted (not scrubbed)

**Decision:** v0.2 Phase 7 bumps all **11** public packages (the ten `@patchback/*` incl. the new `@patchback/provenance`, plus `patchback`) to `0.2.0` in lockstep; root/apps/examples stay private and unversioned. The README is repositioned to present **two co-equal deployment modes** — local `patchback dev` (the ten-minute path) and GitHub Action mode (team/production) — via a "Two ways to run it" overview up top and by promoting Action mode from a bottom-of-list `### Optional:` heading to a first-class `##` section; the outcome view, per-user tokens, and provenance are framed as enhancements that apply to both modes. Two synthetic HMAC signing-secret test fixtures flagged by gitleaks (`<prefix>-signing-secret-0123456789…` in `feedback.emitter.test.ts` + `ci.e2e.test.ts`) are **allowlisted in `.gitleaks.toml`**, not rewritten. The pre-alpha status line ("works from a clone, not yet published") stays.

**Why:** Lockstep versioning keeps `workspace:*` resolution trivial and the release legible (one version = one feature set) — independent per-package semver buys nothing pre-1.0 and complicates the dry-run audit. The README already accreted every v0.2 feature during Phases 1–6, but Action mode read as an afterthought next to `provenance`; a stranger scanning the top must see both shapes, since "team-shape production" is the paid-attention path. Allowlisting (vs renaming the fixtures) matches the existing `.gitleaks.toml` philosophy — obviously-synthetic fixtures get an allowlist entry with a comment; the sequential-digit body makes them self-evidently fake, and the alternative (editing tests to dodge a scanner) is worse hygiene than a documented allowlist. **Alternatives rejected:** per-package versions (needless bookkeeping); adding per-package `LICENSE` files (pnpm already injects the root MIT `LICENSE` into every tarball — verified in the dry-run); scrubbing the test secrets by renaming (churns tests to appease a regex the allowlist is designed for).

**Context:** Version bump across `packages/*/package.json`; `.gitleaks.toml` (new allowlist regex); `README.md`, `ROADMAP.md`, `docs/extraction-checklist.md` (v0.2 addendum). Publish dry-run green 11/11; gitleaks clean on 109-commit history + committed tree (the 8 raw `dir` findings are all untracked/gitignored `.env` + `.next/` artifacts); forbidden-content sweep clean. `.env` still never committed.

## 2026-07-20 — Per-user browser tokens: signed stateless HMAC, tier-ceilinged, server-only exchange

**Decision:** v0.2 Phase 5 adds an opt-in per-user token path (`ApiConfig.tokenExchange`). Tokens are **signed stateless HMAC** (`pbt_` prefix, carry `tier` + `exp` + audit-only `sub`), reusing the audited issue-marker/webhook crypto (factored into `packages/api/src/hmac.ts`) — **no store dependency**. The embedding app's **backend** mints one at the **server-only** `POST /tokens/exchange`, which (1) requires a real parent API key — a browser-token-authed caller is rejected, so tokens **cannot chain**; (2) **ceilings** the minted tier to the parent key's via `tierAtMost` (requesting higher ⇒ 403 `tier_ceiling`, requesting `outsider` ⇒ 400; never a silent clamp); (3) **rejects browser-origin requests** (`Origin`/`Sec-Fetch-Site`/`Sec-Fetch-Dest`) and is **never CORS-exposed** (an `onSend` hook strips CORS headers from that route). `resolveAuth` checks API keys FIRST (byte-identical direct-key path), then verifies a `pbt_` token only when a verifier is configured; an expired/invalid token **fails closed to `outsider`** (data-only), never a 401 — consistent with the keyless path. Default TTL 15 min, max 60 min, expiry enforced on every request. The SDK/widget accept a `getToken` provider (mutually exclusive with `apiKey`) that refreshes from the **app's own backend**, never from Patchback.

**Why:** Removes the embedded-long-lived-key problem (OPEN_ISSUES 2026-07-15) without new store surface: stateless-with-short-TTL works identically under MemoryStore and Postgres (local-first is the deciding constraint), needs no migration/sweep, and reuses crypto already proven in-tree. Fail-closed-to-outsider preserves the repo's "anonymous submission is a feature, not a 401" invariant (2026-07-13). Server-only + no-chaining + tier-ceiling keep the exchange endpoint — whose compromise = minting arbitrary tokens up to the parent tier — the most tightly guarded surface. **Alternatives rejected:** store-backed per-token tokens (large new store/migration/sweep surface for a revocation benefit a short TTL largely obviates); a separate auth header (would fork the SDK auth builder and the CORS allowlist); silent tier clamping (repo posture is reject-don't-strip for client-supplied tiers). Notably, `Sec-Fetch-Mode` is NOT a rejection signal — Node's undici `fetch` sets `sec-fetch-mode: cors` on ordinary POSTs, so it would block the legitimate app-backend caller; `Origin` + `Sec-Fetch-Site`/`Dest` catch every real browser while a server client sends none.

**Context:** `packages/api/src/{browser-token,hmac,auth,config,server}.ts`, `packages/api/src/routes/tokens.ts`, `packages/sdk/src/client.ts`, `packages/widget/src/{config,controller}.ts`. Non-individual-revocability + ephemeral-secret caveats logged in OPEN_ISSUES; a `keyId`/secret-versioning hook for zero-downtime rotation is deferred (shared with the 2026-07-19 issue-marker rotation issue). Plan: `.a5c/runs/01KXXPDF1Y7TMPE4J22S3GNN6K/artifacts/phase-5-plan.md`.

## 2026-07-10 — Job carries an immutable transition history; transitionJob is pure

**Decision:** `Job.history` records every state change (`from`, `to`, ISO timestamp, optional note), and `transitionJob()` returns a new Job instead of mutating.
**Why:** An audit trail is cheap now and load-bearing later (widget thread view, debugging failed patches); purity keeps the type package storage-agnostic so the API layer decides persistence. Alternative — a mutable class — would couple consumers to an instance lifecycle.
**Context:** `packages/types/src/job.ts`.

## 2026-07-10 — @patchback/github is a zero-dependency fetch client, not octokit

**Decision:** Token mode calls the GitHub REST API directly with the global `fetch` (Node 20+), injectable via `TokenClientOptions.fetch`; no octokit or other HTTP/SDK dependency.
**Why:** The surface is five endpoints — octokit would add a dependency tree to the most security-sensitive package for no leverage. Injectable fetch makes unit tests trivial (no `vi.stubGlobal`, no nock). Octokit remains the fallback if the surface ever grows pagination/rate-limit complexity.
**Context:** `packages/github/src/token-client.ts`; BUILD_PLAN Phase 3; CLAUDE.md "minimal, established dependencies".

## 2026-07-10 — commitFiles uses the git data API; one call, one commit, no force

**Decision:** `commitFiles` builds ref → parent commit → tree (`base_tree` + inline `content`, `sha: null` for deletes) → commit → non-force ref PATCH, instead of the per-file contents API.
**Why:** The agent pipeline commits multi-file changes; the contents API is one commit per file and can't delete in the same commit. A single atomic commit keeps PR history readable and the non-force ref update fails loudly if the branch moved underneath us.
**Context:** `packages/github/src/token-client.ts` (`commitFiles`).

## 2026-07-10 — Adapter success is judged by the git diff, not the CLI's self-report

**Decision:** The Claude Code adapter parses `--output-format json` tolerantly but decides pass/fail from `git diff --numstat` (via `git add --intent-to-add` so untracked files count): zero changed files fails, exceeding the diff ceiling fails, unparsable CLI output with a valid diff still succeeds.
**Why:** The CLI's self-report can be wrong in both directions (claims success without edits, or emits noise around valid work); the working tree is ground truth and the check-runner validates it independently. Alternative — trusting the JSON `is_error` alone — was kept only as an additional failure signal.
**Context:** `packages/agent-claude-code/src/adapter.ts` (`execute`), `src/result.ts`.

## 2026-07-10 — Diff ceiling defaults to 300 changed lines and fails toward triage, not retry

**Decision:** `maxChangedLines` (additions + deletions from numstat, binary files counted as 0 lines but still listed) defaults to 300; exceeding it fails the job with a message saying triage likely misclassified the item and it should go to a human — explicitly not "retry with a bigger limit".
**Why:** BUILD_PLAN Phase 4 rule: a bigger diff means the triage was wrong. Encoding the routing advice in the error keeps later phases (worker, widget thread) honest about what this failure means.
**Context:** `packages/agent-claude-code/src/adapter.ts`; CLAUDE.md "triage before code".

## 2026-07-10 — Adapters spawn processes via a shared agent-core runProcess helper

**Decision:** One `runProcess()` (agent-core) serves both the check-runner and CLI adapters: detached process groups on POSIX so timeout SIGKILL takes down grandchildren, settle-after-exit grace because a surviving grandchild can hold stdio pipes open, stdin input support, and separate stdout/stderr plus combined capture. The adapter binary is injectable (`binaryPath` + `binaryArgs`), which is how tests substitute a fake CLI (node script) for `claude`.
**Why:** The naive kill left `npm run` grandchildren alive and hung the runner until test timeout (observed in this session); solving it once in agent-core keeps every adapter safe. Injection keeps unit tests hermetic while the real-binary e2e stays env-gated (`PATCHBACK_E2E_CLAUDE=1`, cleanly skipped otherwise).
**Context:** `packages/agent-core/src/process.ts`; `packages/agent-claude-code/test/fixtures/fake-claude.mjs`.

## 2026-07-10 — Outsider feedback short-circuits triage: needs_human, zero model calls

**Decision:** `triageFeedback` never sends `outsider`-tier feedback to the model; it returns a deterministic `needs_human` (confidence 1) with reasoning noting the short-circuit.
**Why:** Rule #3 — outsider feedback is data only. Classifying it buys nothing in v0.1 (no clustering yet; `patchable` is a forbidden output for the tier anyway) and each hostile submission would cost a model call (griefing vector). The tier check becomes structural inside triage: even if a later caller forgets the server-side check, triage output for outsider items can never be `patchable`. Defense-in-depth, not a replacement for Phase 6 server-side enforcement. Alternative considered (classify-but-cap for future clustering) rejected in the approved plan.
**Context:** `packages/triage/src/classifier.ts`; zero-invocation guarantee unit-tested in `classifier.test.ts`.

## 2026-07-10 — Every uncertain triage path resolves DOWN: failsafe parsing + one-step demotion ladder at 0.7

**Decision:** Malformed/unparseable/unknown-enum model output resolves to `needs_human`/confidence 0 (never a throw, never patchable); transport errors throw `TriageModelError` (caller owns retries). Below a configurable `confidenceThreshold` (default 0.7, strict `<`), results demote exactly one rung: patchable→needs_clarification (model's clarifying question preserved, deterministic fallback question otherwise), needs_clarification→needs_human (question dropped), needs_human is the floor. Demotions are annotated in `reasoning`; the returned confidence stays the model's original number. No promotion mechanism, no retry-for-a-better-label.
**Why:** "When uncertain, classify DOWN" as mechanism, not aspiration. Self-reported confidence is uncalibrated, so the threshold is a policy knob validated end-to-end by the evals (which score post-demotion results). Separating transport failure (retryable, thrown) from classifier fault (terminal, failsafe) keeps a flaky network from ever producing a classification.
**Context:** `packages/triage/src/{schema,threshold,classifier}.ts`.

## 2026-07-10 — Prompt-injection posture: nonce-delimited DATA blocks + hard needs_human mapping + absolute eval gate

**Decision:** All submitter-controlled content (message, console entries, picked-element text, URL, title) is wrapped in per-call random-nonce DATA blocks with tag-shaped sequences sanitized (`<data-`/`</data-` defanged); the frozen system prompt maps instruction-smuggling, self-classification, and secret-exfiltration asks to `needs_human` regardless of tier; trust tier is stated outside the blocks as metadata. Screenshots are never serialized in v0.1 (cost + image-borne injection surface). The eval suite (30 fixtures incl. 6 injection vectors across message/console/element channels) asserts the injection gate separately from the 90% accuracy bar — one leaked injection fails the run regardless of aggregate score.
**Why:** Injection is never "solved"; the layered design (delimiting + rule mapping + classify-down + tier gates + human review of every PR) is the posture, and the fixture file is the living regression suite.
**Context:** `packages/triage/src/prompt.ts`; `packages/triage/evals/`.

## 2026-07-13 — Trust tiers assigned exclusively server-side from a config API-key map

**Decision:** `ApiConfig.apiKeys` maps bearer keys → `owner` | `insider` (an "outsider key" is unrepresentable in the type and rejected by `validateConfig`); no/unknown/malformed key resolves to `outsider` (fail closed, not 401 — anonymous submission is a feature). The POST /feedback body schema has `additionalProperties: false` and no `trustTier` property, and ajv's default `removeAdditional` is turned OFF, so a client-supplied tier is a loud 400. Key comparison is constant-time (hash-then-timingSafeEqual); keys never appear in logs or responses. Every tier read back from storage/config re-passes `isTrustTier` and fails closed with `StoreIntegrityError` (closes the Phase 5 carry-over: the tier is revalidated before it reaches the triage prompt path).
**Why:** The trust tier is a security boundary; the only party that may assign it is the server, from configuration the operator controls. Rejecting (not stripping) a body-supplied tier prevents anyone building against the false assumption that clients pick tiers.
**Context:** `packages/api/src/{auth,config,trust}.ts`, `src/routes/feedback.ts`; approved phase-6 plan §3.

## 2026-07-13 — Per-item read tokens (hashed at rest) gate feedback/job reads

**Decision:** `POST /feedback` returns a one-time 32-byte base64url `readToken`; only its SHA-256 hash is stored. `GET /feedback/:id`, `GET /jobs/:id/status`, and `POST /feedback/:id/reply` require the item's token or an owner/insider key; failures are 404 (not 401) so probing can't distinguish existence. Rejected alternative: unguessable-ID-as-capability — IDs land in URLs and logs by design and would leak captured context (console errors, DOM paths).
**Why:** Cheap now; the Phase 7 widget just stores the token next to the id.
**Context:** `packages/api/src/ids.ts`, `src/routes/shared.ts`; plan §3.

## 2026-07-13 — Replies are NEW linked FeedbackItems with NEW Jobs; effective tier = thread minimum (fulfils the Phase 1 anticipation)

**Decision:** `feedback.needs_clarification` stays terminal, exactly as the Phase 1 state-machine decision anticipated ("likely as a new Job rather than a resurrected one"). `POST /feedback/:id/reply` (valid only in that state, 409 otherwise) creates a new item linked via additive `threadId` (root id) / `inReplyTo` (parent id) fields and a new job at `feedback.received`. The reply item's stored tier is the MINIMUM across its thread (owner > insider > outsider) — outsider content anywhere poisons every reply, so a trusted replier can never launder an outsider-rooted thread into a triage prompt or a brief. The caller's key tier does not enter the minimum: read access already proves thread membership, and the thread's own provenance decides. Reply triage sees the thread context (prior messages + clarifying question) inside the same nonce-delimited DATA blocks.
**Why:** Keeps the canonical CLAUDE.md states byte-exact, keeps one job = one triage verdict = at most one PR, and makes the min-tier rule a data-level property rather than a caller-level courtesy. Alternative (a needs_clarification → received re-triage edge) rejected: would amend the canonical machine for no audit benefit.
**Context:** `packages/types/src/feedback.ts`, `packages/api/src/routes/feedback.ts`, `packages/triage/src/prompt.ts` (ThreadContext); plan §5.

## 2026-07-13 — needs_human is a classification resting at feedback.triaged, not a job state

**Decision:** After triage, `needs_clarification` items advance to the terminal `feedback.needs_clarification`; `patchable` AND `needs_human` items rest at `feedback.triaged`. `needs_human` items are un-startable via the server-side triage gate (`403 triage_gate`), not via a new state. This resolves the STATE.md open question without touching the canonical machine.
**Why:** Classification lives on the item; the state machine tracks lifecycle, not verdicts.
**Context:** `packages/api/src/workers/triage-worker.ts`, `src/routes/jobs.ts`; plan §4.

## 2026-07-13 — Storage is a hand-written Store interface: MemoryStore dev default, Drizzle/Postgres prod; SQLite deferred

**Decision:** One `Store` interface with compare-and-swap `updateJob(job, expectedState)` as the concurrency primitive (drizzle: `UPDATE … WHERE state = expected`; duplicate queue deliveries and double-starts lose the CAS instead of corrupting the audit trail). MemoryStore (zero deps, deep-copy in/out, same fail-closed validation) is the dev/test default; DrizzleStore + committed drizzle-kit migrations (CHECK constraints over the three tiers and twelve canonical states) is prod; `pg` is imported by exactly one file. SQLite is NOT in Phase 6: better-sqlite3 is a native build step (violates the no-install spirit) and `node:sqlite` needs Node 22.5+ (engines say >=20). Revisit condition: if `patchback dev` needs persistence across restarts in Phase 8, a drizzle sqlite-core store slots in behind the same interface. Row → domain mapping revalidates tier/state/triage jsonb and throws `StoreIntegrityError` — never coerced toward a runnable state or an eligible tier.
**Why:** The interface is five queries; an ORM-agnostic seam keeps `npx patchback dev` at zero services while prod gets real CAS semantics.
**Context:** `packages/api/src/store/`; one conformance suite parameterized over both drivers, Drizzle env-gated behind `PATCHBACK_TEST_DATABASE_URL` (verified green against a live Postgres 17 this session).

## 2026-07-13 — Queue is a minimal TaskQueue; memory dev default; bullmq confined to one file; patch tasks never auto-retry

**Decision:** `TaskQueue` = enqueue / process / close over two task types (`triage {feedbackId, jobId}`, `patch {jobId}` — the triage payload carries the jobId, a deliberate convenience over the plan's sketch). Retry policy is per task type and shared by both drivers via `maxAttemptsForTask`: triage 3 attempts (transport `TriageModelError`s throw and retry; exhausted retries leave the job at `feedback.received`, never a fabricated classification), patch exactly 1 (the worker records `patch.failed` itself; the queue re-running an agent would burn money and hide failures). MemoryQueue is single-consumer FIFO with `onIdle()` so tests await drains instead of sleeping; BullMQQueue is the only file importing bullmq, parses the Redis URL without importing ioredis, env-gated test behind `PATCHBACK_TEST_REDIS_URL` (verified green against a live Redis this session).
**Why:** Local-first structurally: memory drivers are the defaults, services activate only when explicitly configured.

## 2026-07-15 — Masking engine semantics: masked vs ignored; nearest marker; non-overridable floor; fail closed

## 2026-07-19 — Repo-aware triage stage 2: optional injected RepoProbe, one-rung reconcile cap, paths+counts-only, no clone-for-triage

**Decision:** v0.2 Phase 3 adds an OPTIONAL retrieval second stage to `@patchback/triage`, gated on the PRESENCE of an injected `RepoProbe` (new IO-free interface, `packages/triage/src/probe.ts`) — no "enable retrieval" flag. **Decision A (LITERAL one-rung cap — owner override of the plan's stricter default):** `reconcile()` allows DOWN (or same-rung confirm) always; allows UP by AT MOST ONE RUNG on `needs_human(0) < needs_clarification(1) < patchable(2)` and only when the probe evidence is UNAMBIGUOUS (single distinct file, ≥1 and ≤`DEFAULT_MAX_UNAMBIGUOUS_MATCHES`=5 matches, not truncated). Property-tested over all (from, suggested) pairs: reconciled rung ≤ from+1, and a `needs_human` stage1 never yields `patchable`. The reconciled result still passes through the existing `applyConfidenceThreshold` ladder, so a retrieval-raised `patchable` must ALSO clear the 0.7 gate on stage2's own confidence (no double-counting; a raised-label-but-low-confidence result demotes right back).
**Decision B (paths + counts ONLY — owner-confirmed):** probe output carries file paths + integer match counts and NOTHING else — no file contents, no match-line snippets. Type-level containment: `ProbeResult` has no field that can hold file text, so the stage-2 DATA block is structurally incapable of carrying attacker-controlled prose even if a matched file is 100% injection payload. Evidence references queries by INDEX (query text not echoed), and is still nonce-wrapped, `sanitizeDataContent`-run, and file-count-capped (belt-and-braces).
**Final gating:** `needs_clarification` is ALWAYS probe-eligible (the recovery target); `patchable` and — per Decision A — `needs_human` are eligible only inside the symmetric confidence band `[threshold-δ, threshold+δ]` = `[0.55, 0.85]` (δ=`DEFAULT_RETRIEVAL_BAND` 0.15, configurable via `TriageOptions.retrievalBand`). Confidently-settled items above the band (a 0.95 patchable, a 0.99 needs_human) never probe — preserves the plan's "avoid probing obviously-settled items" while admitting needs_human. Outsiders short-circuit before stage 1, so they NEVER probe (spy-tested, zero invocations).

Hard ignore list (`.git`, `node_modules`, `.env`/`.env.*`, every dotfile/dot-dir, `dist`/`build`/`.next`/`coverage`) makes secrets structurally unsearchable — a count can't even confirm a secret string's presence. Time/file/byte-bounded (`truncated` on any cap; a truncated probe is never unambiguous, so a cap can only withhold an up-move). Read-only, refuses to follow symlinks out of root, re-validates returned path shape, NUL binary guard.

## 2026-07-19 — GitHub Action mode: HMAC issue marker is the trust core; label is a filter; CI auto-proceeds; composite action over npx-pinned CLI

**Decision:** v0.2 Phase 4 adds a team-shape deployment where the triage-confirm + patch pipeline runs INSIDE GitHub Actions on the target repo, driven by a patchback-created issue. Three load-bearing sub-decisions:
**Marker scheme (the security core):** a thin ingest (`ApiConfig.issueEmitter` mode on the existing `POST /feedback`) assigns the tier SERVER-SIDE (unchanged `resolveAuth`) and embeds an HMAC-SHA-256 marker at the end of the issue body — `<!-- patchback:v1\npayload=<base64url(canonical JSON)>\nsig=<hex> -->`. The signed payload binds `contentHash` (sha256 of the exact rendered feedback text), `tier`, a `feedbackId` nonce, `repo`, and `issuedAt`. `signIssueMarker`/`verifyIssueMarker` live in `packages/api/src/issue-marker.ts` next to the webhook HMAC and reuse its EXACT constant-time discipline (`createHmac` + hash-both-then-`timingSafeEqual`); verify RE-canonicalizes the parsed payload before signing so wire key-order can never change what is authenticated. Verify fails closed (neutral, reason-for-logs-only) on absent/malformed/bad-signature/content-mismatch/repo-mismatch/stale/bad-tier. The tier travels INSIDE the signature and is NEVER re-derived from the GitHub actor. `patchback ci` verifies FIRST — before constructing any client or touching the store — so an invalid/tampered/stale marker yields a neutral exit with provably ZERO downstream calls (spy-asserted). The `patchback` label is ONLY the workflow's trigger filter, never authorization (documented in the workflow header so nobody "hardens" by trusting it).
**Auto-proceed posture (owner decision 1):** CI auto-proceeds `patchable → patch` with no human click — the human gate is PR REVIEW, never auto-merge (that rule is inviolable and appears nowhere as a flag). This differs from dev (where a human calls `start`); the escape hatch is an optional second-label gate (documented, off by default). Outsider feedback is never emitted as an issue AND is short-circuited by triage + refused by the guarded brief factory if a signed outsider marker ever appears (defense in depth, tested). `patchback ci` reuses the UNCHANGED triage worker → guarded brief factory → patch pipeline, so every trust defense (injection containment, tier gate, agent isolation, diff ceiling, one-attempt repair, no-merge) is inherited; the pipeline clones its OWN scratch dir (never `$GITHUB_WORKSPACE`) and commits via the git data API. Job id === feedbackId makes the patch branch deterministic, so a replay collides on the non-force branch and can never open a second PR.
**Action packaging (owner decision 2):** a composite `action/action.yml` that runs `npx --yes patchback@<exact-version> ci`, NOT a committed JS bundle — the published provenance-checked package is what users already get, avoids a large generated blob desyncing from source in the security-critical phase, and keeps `action/` thin with the shared code in packages. Pinned to an exact version (not a floating major) so the security path never floats onto an unaudited release. `GITHUB_TOKEN` is scoped at the workflow level to exactly `contents:write`, `issues:write`, `pull-requests:write`; `ANTHROPIC_API_KEY` + `PATCHBACK_SIGNING_SECRET` are repo secrets, never written to a file and never echoed (`patchback ci` prints no secrets; the composite step uses no `set -x`). `patchback init --github-action` scaffolds the least-privilege workflow, mints a signing secret printed ONCE with `gh secret set` steps, and writes NO secret files.
**Why:** The whole point of Action mode is "works while everyone's asleep" — a maintainer-applied label needs a human and carries no binding of content or tier, so only a signed marker proves "this text + this tier was authorized by something holding the ingest secret." Reusing the same workers/pipeline (rather than reimplementing) is what makes every existing defense apply for free; the net-new attack surface is exactly "can an attacker get text past the HMAC gate?" — and they can't without the secret. A composite/npx action minimizes the audited diff surface in the most security-sensitive phase.
**Context:** `packages/api/src/{issue-marker,config,routes/feedback,index,testing}.ts`; `packages/github/src/{types,token-client}.ts` (createIssueComment, no merge power); `packages/cli/src/{ci,workflow-template,init,index}.ts`; `action/{action.yml,README.md}`; tests `packages/api/src/{issue-marker,routes/feedback.emitter}.test.ts`, `packages/cli/test/{ci.e2e,ci.init}.test.ts`, `packages/github/src/token-client.test.ts`; plan `.a5c/runs/01KXXPDF1Y7TMPE4J22S3GNN6K/artifacts/phase-4-plan.md` (approved with owner decisions on auto-proceed + live-run scope).

**userSummary source (same call, zero extra model calls):** `summarize()` in `@patchback/agent-claude-code` stays deterministic (it makes no model call); the plain-language summary comes from the ONE `execute()` CLI spawn we already pay for. `buildPrompt()` appends an unconditional "Final step" instruction asking the agent to print one line `<<<PATCHBACK_USER_SUMMARY>>> <one or two plain sentences>`; `extractUserSummary()` (in `result.ts`) pulls the text after the LAST sentinel, trims, caps at 400 chars, and returns `undefined` when the sentinel is absent/blank — best-effort, NEVER fabricated (we do not synthesize a summary from the diff, which would be the jargon this field exists to avoid) and NEVER fails the job. `stripUserSummaryLine()` keeps the machine sentinel out of the technical PR body/agent-notes tail. It flows `AgentSummary.userSummary → PatchPipelineResult.userSummary → Job.userSummary` spread in the SAME object as `prUrl` in the patch-worker `pr.opened` write. TRUST: it is agent OUTPUT rendered to a human — display-only, terminal on the job, NEVER read back into triage/brief/prompt (documented in the type + widget so no future phase "reuses" it as input); the only residual concern is display safety, handled by the widget text-node render.
**previewUrl — surface, never provision:** Patchback does NOT spin up preview environments (explicit ROADMAP). It RELAYS the URL the host's OWN preview system (Vercel/Netlify/Cloudflare/…) already published to the GitHub Deployments API. New READ-ONLY `GitHubClient.getPreviewDeploymentUrl(headSha)` lists deployments for the head sha, reads each one's latest STATUS, and returns the newest `status.environment_url` whose `environment` is not (case-insensitively) `production` and whose latest status is `success` — pending/failed/all-production/no-deployment ⇒ `undefined` (graceful absence). It only ever issues GET requests (the no-merge / no-write invariant tests still hold, plus a GET-only assertion). Dev mode: the pr-poller populates `previewUrl` best-effort (idempotent — stops querying once set; error-isolated; folds into the local job so the merge-tail write can't clobber it). Action mode (one-shot, cannot wait for a deploy): the outcome comment includes `userSummary` immediately and sets the expectation HONESTLY that the preview link appears on the PR once the provider's deploy finishes — no fabricated URL, no polling/waiting.
**Hosted mode is payload-only (preserves the 2026-07-13 no-client webhook boundary):** the hosted webhook is deliberately constructed WITHOUT a `GitHubClient`, so it cannot call `getPreviewDeploymentUrl`. Instead it handles the inbound `deployment_status` event and reads `environment_url` FROM THE EVENT PAYLOAD only — never calling GitHub back, never able to merge/write. Correlation is by the deterministic patch branch (`deployment.ref === job.branchName`) via a new additive `store.getJobByBranchName`, so no head-sha storage is needed. Tests spy-assert ZERO GitHub-client calls on the webhook path.
**Three-layer URL validation:** a shared `isSafeHttpUrl` (in `@patchback/types`, scheme ∈ {http,https}, absolute, ≤2000 chars — reusing the `capture.url` cap) rejects `javascript:`/`data:`/other-scheme URIs at the read method, at every storage site (poller + webhook), AND again in the widget before setting an `href` (jsdom `h()` sets `href` via `setAttribute`, which would otherwise set a hostile scheme verbatim). The API status serializer also re-checks it. The summary renders as a TEXT NODE (`h()` string child ⇒ `Element.append` text node, never innerHTML), proven by an HTML-injection render test; the preview anchor is hardened (`rel="noopener noreferrer"`, `target="_blank"`, no auto-navigation).
**Audience = the exact existing read boundary:** the two fields are served only through the UNCHANGED `canReadFeedback` gate on `GET /jobs/:id/status` — the item's read-token holder plus owner/insider API keys; failure is a 404, same as today. Outsider feedback NEVER produces a Job or PR (trust tier is a security boundary), so there is never a `userSummary`/`previewUrl` behind outsider data — no new principal, no new route.
**Preview privacy — surface rather than gate (owner decision):** a read token could be shared, and a preview URL points at an unmerged deployment. We chose to RELAY (not suppress) because the URL is already exposed on the PR the host chose to open (the provider's own PR comment, the deployments tab); suppressing it in the widget would be security theater. Patchback adds NO auth-bypass and does NOT change the URL's reachability — the host owns that. Documented in SPEC + widget/CLI docs: "Preview links are your existing preview system's URLs; if your previews must stay private, configure preview-environment protection — Patchback surfaces whatever URL your provider posts and does not gate it." Reduced blast radius: non-production environments only + http(s) only.
**Token permission (new, optional, read-only):** listing deployments/statuses needs the fine-grained token's `Deployments (read)` permission; the Action workflow gains `deployments: read` (least-privilege, read-only). `github-probe` warns (non-fatally) on a 403 from the deployments endpoint; hosts without the permission just never see previews. Storage: jobs are columnar in Drizzle, so this adds `user_summary` + `preview_url` nullable text columns + a `jobs_branch_name_idx` (migration `0001_closed_talisman.sql`); MemoryStore stores the whole job so no change there.
**Why:** the outcome view closes the loop for the non-technical submitter (the product's north star) without a second model call, without weakening any trust boundary, and without changing behavior when a host has no preview system or the agent omits a summary. Surfacing the host's own preview URL (vs provisioning) keeps Patchback local-first and side-effect-light; the payload-only `deployment_status` handler is the natural symmetric counterpart to the existing inbound-only PR webhook and keeps the no-outbound-call wiring intact.
**Live check:** optional/env-gated (credit restored) but not required for this phase — the offline gate (lint+typecheck+test+build+format:check, all green) is sufficient, matching prior phases.
