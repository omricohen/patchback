- **[2026-07-15] Embedded apiKey confers its tier on every page visitor** — inherent to the design: the widget's `apiKey` is the EMBEDDING APP's key, so anyone who can load an internal page submits at that key's tier (and the key is visible in page source). Accepted for v0.1 with prominent README warnings (@patchback/widget, @patchback/sdk): internal apps behind the app's own authentication only; public pages go keyless (outsider, data-only). Revisit: per-user token exchange (roadmap thought, not v0.1).
- **[2026-07-15] Closed shadow roots are undetectable by the masking engine** — `element.shadowRoot` is null for closed roots, so the planned "host treated as masked in screenshots" cannot be implemented from outside. Current posture is fail-closed by construction: the renderer cannot serialize closed-root content at all (it never reaches the clone), so nothing leaks — but the host's box is also not painted in layer 2. Documented in the widget README. Revisit only if a renderer gains closed-root access.
- **[2026-07-15] snapdom vendors fonts.gstatic.com icon-font URLs** — the bundled renderer carries four hardcoded Material-Icons woff2 URLs it can fetch when a page uses Material Symbols with FILL=1. Suppressed at runtime (`window.__SNAPDOM_ICON_FONTS__` set to empty strings before module eval in `packages/widget/src/capture/screenshot-snapdom.ts`) and pinned by hygiene tests (source scan + IIFE-bundle origin allowlist that requires the suppression marker). Residual: the suppression relies on snapdom's override global staying supported — re-verify on any snapdom version bump (the hygiene bundle test will catch new origins, not a renamed hook; a renamed hook makes the fetch return, so check the changelog).
- **[2026-07-15] snapdom scrolls the page to the top mid-capture** — handled: the widget saves and restores the scroll position around every render (asserted in the browser acceptance suite). Noting it here because it is undocumented snapdom behavior that silently broke geometry once already (see the 2026-07-15 "Screenshot geometry corrected" decision) — re-verify on version bumps.

- **[2026-07-15] Agent-spawn isolation depends on Claude Code CLI >= 2.1 flags** — the adapter's flag layer (`--bare --strict-mcp-config`, see the 2026-07-15 isolation decision) is version-sensitive: an older `claude` binary will reject `--bare` with "unknown option" (loud failure, not silent leakage), and a future CLI could rename it. The env layer (per-job empty `CLAUDE_CONFIG_DIR` + allowlisted spawn env) and the two sweep filters (diffNumstat + pipeline commit path) do not depend on CLI version. Also note: with `--bare` the agent authenticates strictly via `ANTHROPIC_API_KEY` — `claude` keychain/OAuth login is deliberately never read. Re-verify the flag set on CLI major bumps. Lives in `packages/agent-claude-code/src/adapter.ts` (`DEFAULT_ISOLATION_FLAGS`, overridable per adapter instance).
- **[2026-07-15] `testCommands` in patchback.config.ts is informational only** — the pipeline runs the target repo's OWN package.json scripts via the check-runner; the configured commands are recorded from first-run and surfaced in warnings but not executed. Wiring custom check commands through `DefaultPipelineOptions` is a Phase 9/10 decision if real repos need it (e.g. non-npm projects).
- **[2026-07-15] `npx patchback dev` requires publish** — the bin works from the workspace (`node packages/cli/dist/index.js`), but the true cold-start `npx` path only exists after the real npm publish. Phase 10 (2026-07-15) got the publish dry-run green for all 10 public packages; the actual `pnpm -r publish` needs Omri's npm login (item 1 in the extraction checklist's "Remaining — requires Omri" list).
- **[2026-07-15] Concurrent job starts can duplicate GitHub issues (Phase-6 advisory 2, carried)** — POST /jobs/:id/start creates the issue BEFORE the state CAS, so two racing starts can file two issues (only one job proceeds). Non-critical; needs either issue-after-CAS with a compensating state note, or an idempotency key. Lives in `packages/api/src/routes/jobs.ts`.
- **[2026-07-15] Hidden prompt echo suppression relies on readline internals** — secret input on a real TTY suppresses echo by overriding readline's `_writeToOutput` (the widely-used community approach; piped/test input never echoes regardless). If a future Node release removes that internal, secrets would echo locally — re-verify on Node major bumps. Lives in `packages/cli/src/prompts.ts`.
