# Open issues — Patchback

## Open

- **[2026-07-19] Action mode signing-secret custody & rotation** — `PATCHBACK_SIGNING_SECRET` is a SYMMETRIC HMAC key that lives in two places: the ingest and the target repo's GitHub secrets. If it leaks, an attacker can forge markers and get arbitrary text (at any tier) past the HMAC gate to the agent — the whole Action-mode boundary rests on this key. Mitigations in place: repo secret only (never a file, never echoed; `patchback ci` prints no secrets, no `set -x`), `init --github-action` prints it once, and the freshness window bounds exposure. No automated rotation yet: rotation is manual and must update BOTH sides together (re-`gh secret set` + reconfigure the ingest), and there is no key-id/versioning in the marker to support overlapping keys during a rotation. Lives in `packages/api/src/issue-marker.ts` (marker), `packages/cli/src/{ci,init}.ts`. Revisit: add a `keyId` to the payload + accept a small set of active keys to make zero-downtime rotation possible.
- **[2026-07-19] Action mode replay bound is a window + idempotent branch, not a ledger** — an attacker who can READ a validly-signed issue can copy its marker+body into a NEW issue. This can at worst re-run ALREADY-AUTHORIZED feedback (a duplicate or no-op PR that still needs human review); it can NEVER smuggle new instructions (no signing secret). Layered mitigations: `repo` binding (no cross-repo replay), `contentHash` binding (marker invalid against different text), `issuedAt` freshness window (default 24h, `DEFAULT_MARKER_FRESHNESS_MS`), and a deterministic idempotent branch (`patchback/job-<feedbackId>`, non-force `createBranch`) plus workflow `concurrency` so a same-`feedbackId` replay collides and produces no second PR. There is NO server-side nonce ledger (Action mode is stateless), so a replay with a DIFFERENT issue number inside the freshness window that has not yet been processed could still re-open one duplicate PR before the branch collision — bounded and human-reviewed, called out honestly rather than overclaimed. Lives in `packages/api/src/issue-marker.ts`, `packages/cli/src/ci.ts`. Revisit: optional ingest-side issued-`feedbackId` ledger if the ingest gains persistence; tighten the window per deployment.
- **[2026-07-19] Action mode CI-minutes / DoS on public repos** — the workflow fires on EVERY labeled issue, and anyone who can apply the `patchback` label starts a run — even though a run with no valid marker neutral-exits with zero agent/GitHub-write calls. On a public repo this lets a labeler spin up (cheap, early-exiting) workflow runs. Mitigations: the `if: label == 'patchback'` filter, the neutral early-exit (verify happens before any client/model/agent work), `concurrency` per issue, and `timeout-minutes`. Documented posture: Action mode targets team/internal repos (consistent with the product's internal-apps positioning). Lives in `packages/cli/src/workflow-template.ts`. Revisit: a stronger pre-agent gate (e.g. require the ingest bot as issue author via a workflow `if:`) if public-repo abuse shows up.
- **[2026-07-19] Repo-aware triage stage 2 — working-copy skew (advisory limitation)** — the CLI `LocalRepoProbe` searches whatever is on disk at `localRepoPath`, which may be on a different branch or dirty relative to the base branch the agent later clones. So a probe match count can be stale w.r.t. the code the agent will actually edit. Accepted by design: the probe is an ADVISORY triage signal into a human-reviewed pipeline (triage-before-code, no auto-merge), so skew can only mis-rank a borderline item (or withhold an up-move), never bypass review or launder an injection into `patchable`. Noted in code comments (`packages/cli/src/repo-probe.ts`, `dev.ts`) and the CLI banner. Revisit if hosted/indexed repo-aware triage happens (would want the probe pinned to the base commit).
- **[2026-07-20] Preview surfacing needs the optional `Deployments: read` token permission** — `getPreviewDeploymentUrl` reads the GitHub Deployments API, which a fine-grained PAT can only see with the (new, optional) `Deployments (read)` repository permission; the Action workflow needs `deployments: read` in its `permissions` block. Without it previews simply never appear (graceful absence) — every other feature works. `github-probe` warns (non-fatally) on a 403 from the deployments endpoint, but fine-grained tokens do not expose per-permission scopes for a definitive check, so the warning is best-effort (a repo with zero deployments returns 404/200 and produces no signal). Lives in `packages/github/src/token-client.ts`, `packages/cli/src/{github-probe,workflow-template}.ts`. Revisit if a more reliable permission probe becomes possible.
- **[2026-07-20] Preview provider environment-naming variance** — selection surfaces the newest successful deployment whose `environment` is not (case-insensitively) `production`. Exotic providers that label previews `production`, leave `environment` blank, or use a bespoke prod name would be mis-surfaced or missed. Mitigated by newest-successful-preview selection + the non-prod filter + `isSafeHttpUrl`; documented as a known simplification. Lives in `packages/github/src/token-client.ts` and the webhook/poller callers. Revisit with a configurable production-environment name list if real providers need it.
- **[2026-07-20] Shared read token exposes an unmerged preview deployment** — `previewUrl` is served to the item's read-token holder (and owner/insider keys) and points at a deployment of an unmerged change. A leaked/shared read token therefore leaks the preview URL. Boundary (documented in SPEC + widget/CLI docs, and the 2026-07-20 DECISIONS entry): Patchback only RELAYS a URL the host's own preview system already published to the Deployments API and does NOT gate it — if previews must stay private, the host configures preview-environment protection (e.g. Vercel deployment protection). Blast radius reduced by non-production-only + http(s)-only. Lives in `packages/api/src/routes/jobs.ts` (serialize gate), `packages/widget/src/ui/thread.ts` (render gate). Revisit only if Patchback ever provisions its own previews.
- **[2026-07-20] Hosted preview surfacing depends on the host subscribing `deployment_status`** — hosted mode populates `previewUrl` from the inbound `deployment_status` webhook event (payload-only; the webhook has no GitHubClient and never calls GitHub back), correlating by the deterministic `deployment.ref === job.branchName`. A host that does not subscribe that event, or whose provider does not emit it, gets no hosted previews (dev-mode poller and Action-mode honesty are unaffected). Lives in `packages/api/src/routes/webhooks.ts` + `store.getJobByBranchName`. Revisit with an (opt-in) hosted background preview poller if event coverage proves insufficient (ROADMAP).

- **[2026-07-20] Browser tokens are not individually revocable before TTL** — the per-user token (v0.2 Phase 5) is signed stateless HMAC, so there is no per-token kill switch. Per-token hard revocation would need the store-backed variant (deliberately not built). Lives in `packages/api/src/browser-token.ts`. Revisit: a `keyId`/secret-version field for zero-downtime rotation (shared hook with the 2026-07-19 issue-marker rotation issue), behind which a store-backed per-token revoke could later hang if a deployment needs hard revoke.
- **[2026-07-20] Ephemeral exchange secret ⇒ tokens die on restart / need an explicit shared secret for multi-instance (advisory)** — when `tokenExchange.signingSecret` is omitted, the server generates a per-process secret at startup (dev-friendly: keeps `patchback dev` zero-config) and logs a warning. Consequence: minted tokens do not survive a restart, and in a multi-instance deployment each instance rejects the others' tokens. Set an explicit `signingSecret` for production / horizontal scaling. Lives in `packages/api/src/routes/tokens.ts` (`resolveTokenExchange`), `packages/api/src/config.ts`.
